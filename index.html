    DirectX Group Project

    A short puzzle game where the player manipulates a cubes properties to complete the level.

    The properties you can modify are the material, size, bounciness and conductivity. Each resulting in the cube behaving differently.

    It has basic physics and collisions for the cubes, with properties like mass being changed by its material and size which also affects its other properties like how well it can bounce.

    The game has both 2D and 3D sounds setup using a singleton class and only loads the sounds it needs for each level using the level system which utilises multithreading.

    There is also a fully functioning UI system for the main menu, pause menu, the settings and the HUD. The settings make use of JSON files to read and save all the settings.

    We also had an event system to handle many of the events throughtout the game and to keep the code more organised while not linking multiple systems together.

    DirectX AI Pathfinding Framework

    A framework created with DirectX 11 that uses pathfinding to guide a racecar around a track.

    Uses the A* algorithm with a map of waypoints to create a path for the car to travel. Red waypoints are on the track and have a low travel cost, blue are off the track and have a high travel cost and yellow are checkpoints which the car travels to.

    The car uses a finite state machine to change between different steering behaviours which are: seek, flee, arrive, pursuit, wander and laps.

    There is also a second car which can be toggled (and is needed for flee and pursuit) which the main car has some basic decision making to try and avoid colliding with.

    When in steering behaviours that do not require input from the user left clicking will instead place "oil" on the nearest waypoint, increasing the travel cost to it.

    A powerup is also present that can be respawned and gives the waypoint it is in a lower travel cost. It gives the car a temporary speed boost.

    If I was to attempt this again I would add improved steering to the car instead of it just snapping to a direction and also improve the decision making as it is currently very basic.

         DirectX Physics Framework

    A framework created with DirectX 11 which has basic collisions with 3D objects.

    It has a free roam camera that can be toggled to have gravity and collisions with the surface below it.

    Every moveable objects can have forces applied to them like gravity, friction, drag and thrust with each object having their own properties allowing for basic physics.

    There is a "vehicle" which can be steered using the direction of the wheels or it can hover and fly, depending on the mode selected.

    Each object also has their own hitbox to detect and handle collisions for example moving and rotating (via quaternions) the object which is hit or when an object is on solid ground.

    A particle system has also been implemented to allow for particle effects which can be manipulated and they can be reused for different particles for better memory usage.

    If I were to do this project again I would update the hitboxes to be more accurate and allow for the hitboxes to rotate with the object and to implement more realistic physics.

    DirectX Framework

    A framework created with DirectX 11 in which the user can create and transform different textured 3D objects.

    The user can move around a 3D space and look around with basic mouse input with the option to switch between different cameras, which are each stored in an array, for different perspectives.

    Pressing the left mouse button creates textured objects in front of them which are loaded from OBJ and DDS files. The object that is loaded can be changed and so can its material.

    Pixel shading is implemented with the use of a shader file which is affected by the object material and the colour of the light can be changed as well.

    On this project if I was to attempt it again I would improve the rotation of the main camera as the current implementation is very basic and not great to use. Also to add more features like creating and deleting different lights and to restructure the code into more classes for better a better layout and control.

         Network Application

    A chat application that sends packets to and from a server to multiple clients and utilises multithreading.

    Using Windows Forms I created a basic GUI which easily allows the user to send packets containing messages to all the other clients or a specific client.

    The server and each of the clients run on their own threads to reduce concurrency issues like freezing until a server response is received and it helps improve performance.

    There is a paint window which allows the clients to paint on the same canvas by sending packets with which pixels coordinates had been drawn on and with what RGBA values to use.

    If I was to do this again I would move packet handling to the client instead of the server and then send off more types of packets depending on what they are needed for like private or encrypted messages.

    Summer SDL Project

    A simple SDL game I made in the summer of 2020 which has no objectives but was used to experiment with SDL.

    The levels are generated by taking the pixel colour of each pixel from the map's PNG file and converting it into a bitmap. If the pixel colour matched the requirements for a tile that tile would be placed in the corresponding place in the game.

    By also learning from an online tutorial it helped me create an Entity Component System which helped with moving the sprites around with the camera and adding properties like collision to the water tiles.

    The camera edge collision was done by creating a rectangle the size of the viewport, the rectangle would detect when it has gone outside the specifed boundaries and stop the camera at the edge of the screen.

    The pause and options menu was the first time I had tried creating my own user interface and I added basic mouse support by creating a mouse collision function that detected when the mouse was over a highlighted option.

    In the case that I do something like this again I would improve how the map tiles are handled as in the current system you can only have so many textures per level and level handling is not done in a very efficient or clean way.

         Asteroid Game

    A 3D game made in OpenGL where you get to control a space ship in an asteroid field.

    This was my first 3D project and it really helped me understand the basics of OpenGL and creating a game in a 3D space.

    There are a set amount of asteroids in the game and when they go behind the camera they are translated back to the front of the game and with the use a random number generator are given random X and Y coordinates.

    Collision detection was done by taking the position of the player and each asteroid and using 3D Pythagoras' thereom to calculate the distance between them and if it was smaller than the radius of the ship it would class as a collision.

    In the future if I try something like this again I might try adding in better UI instead of a menu image and text with a 3D position which caused lighting to effect it. Also to also split the code into more classes as it mostly in the main class.

    Mario Game

    A short and simple Mario game made in SDL consisting of two levels and two players which are both controlled using a single keyboard.

    The collision for the terrain in each level is determined by a simple bitmap and the player sprite positions are checked against it to see if they are touching something which should be solid or not and then acts accordingly.

    This project is the first time I had to think about changing levels so I chose to recreate going down a pipe by detecting whether a player was above it and then moved them down the screen and used a Y-coordinate check on the sprites to switch the level.

    If I was try the project again I would redo how textures are loaded and deleted as they currently cause a small memory leak, probably by loading the assets initially instead of loading them each time they are used.

    Pacman Game

    A simple game with the theme of Pacman but plays slightly differently.

    The 50 munchies are spread out randomly and are put through a for loop to detect if any overlap and if any do they are moved to another location on the screen.

    There is a box that follows the mouse, the box can detect collisions via AABB collision, if the box touches a ghost the ghost is sent in the opposite direction, and while the box collides with Pacman it disappears.

    If you pick up the skull the ghosts change behaviour, instead of travelling up or across they instead move towards Pacman by comparing their X and Y coordinates with Pacman's and move accordingly.

    If I was to attempt this project again I would make it object orientated instead of having everything inside of one class as it was my first game in C++.
